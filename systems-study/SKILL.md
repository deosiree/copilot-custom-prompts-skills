# 目标
围绕“结构化系统思维学习法”这一类任务，稳定执行原 prompt 想表达的核心能力，并把零散命令沉淀为可复用的 skill。

## 适用场景
- 当需要在特定应用背景下系统化学习一个新知识点，并从本质、选型、指标、工程演进四个层面建立完整认知框架时使用。
- 适合需要复用同一类提示词、同时希望 agent 按固定结构稳定输出结果的场景。

## 输入要求
- 与该 prompt 对应的原始输入材料

## 输出要求
- 输出需为 Markdown 结构，保持标题、列表、代码块等格式要求
- 若要求生成 Mermaid，必须只输出合法的 Mermaid 代码块

## 执行步骤
1. 先识别输入材料、上下文来源和隐含占位符，明确本次处理对象。
2. 提取原 prompt 的核心任务、结构约束和禁止项，再将它们转写为稳定的执行顺序。
3. 在生成图表前先判断最合适的 Mermaid 图类型，再按语法限制产出代码块。
4. 在最终输出前逐项校验格式、语气、长度和保留项，避免漏掉原 prompt 的关键约束。

## 保留约束
以下原始约束在执行时必须继续遵守：

```markdown
# Role
你是一位精通系统工程与架构设计的专家，擅长从底层协议到上层应用的级联拆解，能够精准识别任何技术栈的“关键性能点”与“博弈权衡”。

# Context
我需要学习一个全新的知识点：【在此输入知识点名称】。
请将其置于【在此输入应用背景，如：某开源项目、特定业务挑战】的上下文中进行分析。

# Output Framework (Strictly Markdown)

## 1. 概念解构：本质与边界
- **第一性原理定义**：剥离所有修饰词，该知识点的最本质物理/数学/逻辑基础是什么？
- **能力边界**：它“能做什么”以及绝对“不能做什么”？（明确其非目标 non-goals）。

## 2. 范式对比：选型辩证法
- **参照系选择**：找出 1-2 个与该知识点在功能上重叠、但实现哲学完全不同的替代方案（如：Grep vs. RAG, SQL vs. NoSQL）。
- **博弈分析**：
    1. **为什么选择 [当前知识点]**：在特定上下文下的核心获益。
    2. **机会成本**：为了选择它，我们在复杂度、资源或一致性上做出了哪些牺牲？

## 3. 度量衡构建：定义其“第一性指标”
- **核心评价体系**：请根据该知识点的特性，提炼出评估其优劣的 **3个关键维度（核心指标）**。
- **链路实现演示**：针对上述每个维度，提供一个**实战场景/代码片段**，并解释数据是如何在系统中流转并达成该指标的。

## 4. 工程演进：从 Demo 到生产
- **关键链路图示**：用文本/Mermaid 形式描述其从输入到输出的标准逻辑链路。
- **失效模式分析**：在极端压力或异常场景下，该知识点最先崩溃的点在哪里？如何加固？

---

# Constraints
- **禁止泛泛而谈**：如果指标是“快”，必须说明是“响应延迟”还是“吞吐量”，并给出量级预估。
- **代码驱动**：所有原理说明必须辅以最简化的逻辑代码或伪代码。
- **动态适配**：根据知识点的领域（算法、工具、协议或框架），动态调整评价维度的命名。
```

## 注意事项
- Mermaid 场景优先保证语法正确，其次再考虑节点命名美观。
